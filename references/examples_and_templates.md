# Examples and Templates Reference

Use these as skeletons only. Replace `X/Y/A/B` with material extracted from the current source.
Do not copy domain vocabulary from these examples unless the current author actually uses it.

## 1. 开头模板

### 反差型

```text
我原本以为，做 XX 最难的是 A。
真正做起来才发现，真正难的是 B。
```

### 失败型

```text
第一次做 XX 的时候，我犯了一个很典型的错误：我把问题理解成了 A，但它本质上其实是 B。
```

### 场景型

```text
有一天我在 [具体场景] 的时候，突然发现一个很尴尬的问题：[表面上看起来成立的事]，在真实情境里并不成立。
```

### 判断型

```text
如果现在让我重新经历一遍 [具体事件]，我不会先做 A，而是会先确认 B。
```

## 2. 标题模板

```text
我原本以为 X，后来发现真正重要的是 Y
做完 XX 后，我重新理解了 Y
如果你也在做 XX，先避开这几个坑
从 XX 到 XX：我踩过的坑和后来的判断
普通人做 XX，真正拉开差距的是这件事
我经历了 XX，但最大的收获不是表面上的 Y
```

## 3. 小标题模板

```text
一开始，我把问题想简单了
真正卡住我的，不是 A，而是 B
我踩过的 3 个坑
后来我换了一种处理方式
这套方法可以怎么复用
它不适合所有人，这里有几个边界
```

## 4. 方法组件模板

### 三步法

```markdown
## 我的 3 步方法

1. 先确认目标：这件事真正要解决什么？
2. 再拆关键动作：哪些环节决定结果？
3. 最后定边界：什么情况下这套方法不适用？
```

### 避坑表

```markdown
| 常见坑 | 表现 | 根本原因 | 修正方法 |
|---|---|---|---|
| 把问题看成 A | 表面动作很多，结果仍然不稳定 | 真正变量其实是 B | 先确认 B，再决定动作 |
```

### 检查清单

```markdown
## 发布前检查

- 这篇文章有没有一句核心观点？
- 开头 300 字有没有冲突？
- 有没有具体案例而不是空泛建议？
- 有没有读者可以复用的清单/模板/判断规则？如果用了表格，它是否真的比自然段更清楚？
- 有没有说明方法边界？
```

## 5. 结尾模板

```text
这件事给我最大的提醒是：XX 从来不只是表面上的 A，它背后真正起作用的是 B。
```

```text
如果你也刚开始做 XX，不必一开始就追求完整答案。先找到那个最小但真实的场景，再从里面提炼自己的判断。
```
